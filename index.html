AI Automation Analysis

What we are actually trying to build

The immediate objective is not “AI for everything.” The realistic first milestone is a coding agent that understands SyenApp’s stack, patterns, and quality expectations. After that, the system can expand into UI, analytics, support, and marketing use cases.

Phase 1 targetCoding agent for Node.js, React Native, and project-aligned Python support
Key principleUse current architecture first; suggest improvements separately
Knowledge modelRAG with existing Elasticsearch plus a dedicated AI knowledge index
Rollout modelSingle agent first, multi-agent later
Implementation options

Ways to build and access the agent

Option 1

Shared GPT / Claude Project

Fastest path to evaluation. Good for uploading knowledge, testing prompts, and comparing model quality with minimal engineering work.

Option 2

Custom UI backed by API

Internal portal with role-based access, logging, prompt templates, and room for future approvals or workflow controls.

Option 3

IDE / Extension access

Best developer experience. The agent can be exposed to VS Code users through an internal service or compatible extension workflow.

Option 4

Full custom agent platform

Most scalable long-term option, but it is too large for the initial SyenApp phase unless single-agent quality is already proven.

Why not one agent for everything

Single-agent limitations

A single all-purpose agent sounds efficient, but in practice it mixes conflicting rules, context, and quality standards across very different departments.

Clear example: if the same agent is asked to generate a Node.js API, draft a push campaign, and answer a reward-ticket dispute, it must switch among engineering, marketing, and support constraints. That makes prompts heavier and outputs less reliable.

RAG strategy

How project knowledge should work

  • Use the existing Elasticsearch cluster as the base data provider.
  • Create a dedicated knowledge index such as syenapp-ai-knowledge for code snippets, docs, schemas, and implementation references.
  • Do not mix raw analytics/event data directly into prompts without curation.
  • Start with keyword retrieval if needed, then add vector search only when relevance requires it.
  • Alternative RAG options remain open: PostgreSQL + pgvector, Pinecone, Weaviate, Azure AI Search, AWS Bedrock.
Important clarification: Elasticsearch for RAG is an extension of the current setup, not a replacement. Existing analytics usage continues; the new knowledge index simply makes the data retrievable for AI.
